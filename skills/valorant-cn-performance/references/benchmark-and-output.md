# 基准、回滚与交付

## 最小基准

记录：

```text
日期/游戏补丁：
服别与地图/模式：
分辨率/显示模式/画质预设/帧率上限：
Windows 电源状态与后台应用：
采样时长：
avg FPS：
1% low FPS：
帧时间异常：
RTT/丢包（网络症状时）：
体感备注：
```

优先使用游戏内统计。需要更稳定的帧时间数据时，可让用户自行选择 PresentMon 或 CapFrameX；说明安装来源、权限与采集开销。

## 实验规则

1. 热身后在同一场景采样，至少重复 3 次。
2. 每轮只改一类；补丁、驱动、地图或后台环境变化就不能直接横比。
3. 同时报中位结果与波动，不只挑最好的一次。
4. 没有可重复收益就回滚。体感变化与指标冲突时标记不确定，继续区分实验。

## 回滚卡

每项条件改动先写：

```text
改动：
原值/截图位置：
触发条件：
风险与依赖：
期望观察：
复测窗口：
回滚步骤：
```

不要假定系统还原已启用；还原点只能作为可用时的补充，不能替代逐项记录。

## 交付模板

```markdown
# 诊断方案（没有 A/B 数据前不得写“优化完成”）

## 结论与置信度
最可能路径：……；置信度：高/中/低。

## 已知 / 缺失
- 用户确认：……
- 只读脚本观察：……
- 仍缺：……

## 症状分类
客户端 FPS / 网络 / 显示与输入 / 反作弊启动 / 混合。

## 优先动作
1. 动作；依据等级；适用条件；期望观察；回滚。

## 暂不建议
- 做法；不做的原因。

## 复测协议
- 同场景、同设置、三轮：avg FPS / 1% low / 帧时间 / RTT / 丢包。

## 来源与日期
- 直接 URL；发布日期/访问日期；适用范围。
```
